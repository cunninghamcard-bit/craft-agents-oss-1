# 业务输出格式

最终回复给用户时只用业务语言，不写搜索和 agent 执行细节。

采购判断本身的结果细节要保留，包括型号差异、资料来源、缺少的材料、供应商需要补什么。

不要把内部执行信息写给用户，包括：

- 资料检索的方法名、步骤名或内部分类名。
- 脚本、模型、prompt、cwd、skill、agent 等系统信息。
- 额度、子任务失败、执行日志等与采购判断无关的信息。

使用下面格式：

```text
结论：不影响使用 / 影响使用

不足类型：无 / 差异资料不足 / 关系证据不足 / 使用条件不足

一句话理由：...

处理建议：可接收报价型号 / 暂不接收，需补材料 / 不接收报价型号 / ...

型号对比：
- 采购型号：...
- 报价型号：...
- 差异：...

查到的资料：
- 来源：...
  链接：...
  说明：这份资料能证明什么。

没查到的关键资料：
- ...

为什么影响判断：
- 说明当前缺口为什么导致不能接受报价型号；如果不足类型是“无”，写“无”。

遇到的阻碍：
- 页面打不开、网站要求验证、资料暂时无法访问时写清楚；没有就写“无”。

还需要补充的材料：
- ...

置信度：高 / 中 / 低

说明：
- 本判断只覆盖型号差异是否影响使用；不覆盖供应商授权、真伪、库存、价格、交期、包装来源风险，除非本次资料明确涉及。
```

不足类型定义见 [decision-rules.md](decision-rules.md) 的"不足类型"和"下一步动作"章节。使用规则：

补材料要具体，不能只写“人工确认”。

可执行的补材料例子：

- 要原厂替代说明。
- 要订货编码表或 part numbering 说明。
- 要原厂图纸或 datasheet。
- 要铭牌照片或原包装标签。
- 要供应商/原厂书面确认。
- 要授权证明或 COC，但只在来源风险和本次判断有关时写。
